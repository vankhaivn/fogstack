What it is

AWS-shaped endpoints, on your machine

fogstack gives a personal project a Kubernetes cluster, an image registry, a Postgres database, and optionally an AWS-compatible API and OpenSearch — so you can develop and wire integrations without spending cloud money or touching a real account.

The one rule
A fogstack command must never read or write your ~/.kube or ~/.aws, and never reach a real AWS account. Everything else in the design follows from this.
A local development tool — not a production-parity environment.

Host isolation · the heart of the design

Every client is redirected away from your host

Before any command runs a kubectl, aws, helm, or terraform client, a guard rewrites where each one looks:

KUBERNETES

KUBECONFIG → .state/

Points at a repo-local kubeconfig. Your host context is never read or changed — it never runs kubectl config use-context.

AWS CONFIG FILES

config & credentials → .state/

AWS_CONFIG_FILE and AWS_SHARED_CREDENTIALS_FILE point inside .state/, not at ~/.aws.

CREDENTIALS

fake, local-only

Exports test / test, region us-east-1, for the process. The instance-metadata endpoint is disabled.

REAL-ACCOUNT VARS

removed

AWS_PROFILE, AWS_SESSION_TOKEN, AWS_ROLE_ARN are unset. fog doctor fails if a profile is still set.

Host isolation

Robust, not best-effort

01 · EVERY ENTRY POINT

The guard runs everywhere

It is not only in the main fog script — the scripts that create the cluster and start the load balancer source it too. No code path reaches a client with host configuration active.

02 · AFTER .ENV LOADS

The guard is re-asserted

The fog script loads your .env, then sources the guard again. Even a .env that tried to set KUBECONFIG or an AWS path cannot redirect a client back at your host files.

Request flow

How a command reaches a local endpoint

1

fog up

Starts or reuses the containers for the chosen profile and waits for health.

2

eval "$(fog endpoints)"

Exports the connection variables into your current shell.

3

your tools connect

kubectl, helm, psql, aws, and Docker use those variables to reach the local containers.

note — AWS SDKs and the CLI must be given the endpoint explicitly (--endpoint-url "$AWS_ENDPOINT_URL"); they do not discover it. Variables are per-shell, so re-run the eval in each new terminal.

Networking

Three wiring details worth knowing

IMAGES → CLUSTER

You push to localhost:5001. The registry also sits on the kind network, and each node has a containerd mirror — so nodes pull the same tag over that network.

AWS API FROM PODS

A pod calls fogstack-emulator:4566 directly — no host round-trip. The emulator holds the Docker socket to launch its own backing containers. OpenSearch is bridged the same way; Postgres & Dashboards are host-only.

LOAD BALANCER

cloud-provider-kind watches LoadBalancer Services and runs one LB container each, publishing a localhost port. On Docker Desktop that published port is the reliable way in.

The published localhost port matters because the assigned ingress IP is often not directly routable from the host on Docker Desktop.

Profiles

minimal by default, full when you need AWS

MINIMAL · the common case

Kubernetes, registry, database

The set most development needs, and it starts quickly. This is what fog up launches by default.

FULL · when AWS is in play

+ AWS API, OpenSearch, Dashboards

Adds the AWS-shaped services, which cost more memory and start time. Switch with --profile full.

why split — keeping the common case fast, and never advertising dead endpoints: fog endpoints deliberately omits AWS_ENDPOINT_URL in minimal mode.

Boundaries

What it enables — and what it doesn't

ENABLES
  • A fast, disposable, AWS-shaped local environment
  • Safe to run next to real company credentials
  • Development feedback and integration wiring
  • Learning the shape of AWS-adjacent workflows
DOES NOT
  • Give production parity
  • Implement every AWS service to the same depth
  • Enforce IAM as a real authorization boundary
  • Enforce real network policy
The AWS-compatible emulator is young software and can change faster than AWS itself.
