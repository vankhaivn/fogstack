# AWS Recipes

Short, copy-paste workflows for the AWS-compatible endpoint in the full
profile. Each recipe assumes the stack is up and endpoints are exported:

```bash
./engine/fog up --profile full
eval "$(./engine/fog endpoints --profile full)"
```

Every command targets `$AWS_ENDPOINT_URL` with fake local credentials. Nothing
here can reach real AWS. Remember the platform limits: IAM is not a real
authorization boundary, and the emulator is young software — if a call fails
unexpectedly, check `docker logs fogstack-emulator` before assuming your code
is wrong.

## S3: Buckets And Objects

```bash
aws --endpoint-url "$AWS_ENDPOINT_URL" s3 mb s3://my-bucket
echo "hello fogstack" > /tmp/hello.txt
aws --endpoint-url "$AWS_ENDPOINT_URL" s3 cp /tmp/hello.txt s3://my-bucket/hello.txt
aws --endpoint-url "$AWS_ENDPOINT_URL" s3 ls s3://my-bucket
aws --endpoint-url "$AWS_ENDPOINT_URL" s3 cp s3://my-bucket/hello.txt -
```

Presigned URLs work and resolve to localhost:

```bash
aws --endpoint-url "$AWS_ENDPOINT_URL" s3 presign s3://my-bucket/hello.txt
curl -fsS "<printed URL>"
```

SDK note: configure path-style addressing (`s3_use_path_style` /
`forcePathStyle`), because `my-bucket.localhost` virtual-host addressing does
not resolve. The override generated by `fog tf-init` already sets this for
Terraform.

## SQS: Queue, Send, Receive

```bash
aws --endpoint-url "$AWS_ENDPOINT_URL" sqs create-queue --queue-name jobs
QUEUE_URL="$(aws --endpoint-url "$AWS_ENDPOINT_URL" sqs get-queue-url \
  --queue-name jobs --query QueueUrl --output text)"
aws --endpoint-url "$AWS_ENDPOINT_URL" sqs send-message \
  --queue-url "$QUEUE_URL" --message-body '{"job":"resize","id":1}'
aws --endpoint-url "$AWS_ENDPOINT_URL" sqs receive-message \
  --queue-url "$QUEUE_URL" --wait-time-seconds 5
```

Delete a received message with its `ReceiptHandle`, or wipe the queue while
iterating on a consumer:

```bash
aws --endpoint-url "$AWS_ENDPOINT_URL" sqs purge-queue --queue-url "$QUEUE_URL"
```

A local worker process consumes the queue the same way: point the SDK client
at `$AWS_ENDPOINT_URL` with the fake credentials from
[Connect Your Project](connect-your-project.md) and poll `ReceiveMessage`.

## IAM And STS: Development Flows

Create-and-read flows work, which is enough to exercise application code that
manages users, roles, and policies:

```bash
aws --endpoint-url "$AWS_ENDPOINT_URL" iam create-user --user-name app-user
aws --endpoint-url "$AWS_ENDPOINT_URL" iam list-users
aws --endpoint-url "$AWS_ENDPOINT_URL" sts get-caller-identity
```

Do not test security assumptions here: policies are accepted but not enforced
as a real authorization boundary.

## Lambda: Quick Check

Lambda support runs function containers through the emulator's Docker access,
so the first invocation pulls a runtime image and is slow.

```bash
cat > /tmp/handler.py <<'EOF'
def handler(event, context):
    return {"ok": True, "echo": event}
EOF
(cd /tmp && zip -q function.zip handler.py)

aws --endpoint-url "$AWS_ENDPOINT_URL" lambda create-function \
  --function-name echo \
  --runtime python3.12 \
  --handler handler.handler \
  --zip-file fileb:///tmp/function.zip \
  --role arn:aws:iam::000000000000:role/lambda-dev

aws --endpoint-url "$AWS_ENDPOINT_URL" lambda invoke \
  --function-name echo \
  --payload '{"name":"fogstack"}' \
  --cli-binary-format raw-in-base64-out /tmp/out.json
cat /tmp/out.json
```

The role ARN is not validated locally. If `create-function` or `invoke`
fails, check `docker logs fogstack-emulator`; Lambda is one of the faster
moving parts of the emulator.

## Reach The Endpoint From Inside The Cluster

Pods in the kind cluster can call the same emulator without going through the
host. The emulator container is attached to the cluster's Docker network under
the name `fogstack-emulator`, so from a pod:

```bash
kubectl --kubeconfig "$KUBECONFIG" --context "$KUBE_CONTEXT" run aws-probe \
  --rm -it --restart=Never --image=amazon/aws-cli:2.34.20 \
  --env AWS_ACCESS_KEY_ID=test --env AWS_SECRET_ACCESS_KEY=test \
  --env AWS_DEFAULT_REGION=us-east-1 -- \
  --endpoint-url http://fogstack-emulator:4566 s3 ls
```

For application manifests, set the endpoint to
`http://fogstack-emulator:4566` (or front it with an `ExternalName` service in
your namespace if you prefer a cluster-local DNS name).

## Terraform Against The Local Endpoint

Generate a local provider override in your project's Terraform directory:

```bash
./engine/fog tf-init ../my-app/terraform
```

The generated `fogstack_override.tf` wires the AWS provider to the local
endpoint for S3, SQS, IAM, STS, RDS, and EKS, with a local backend and test
credentials. Then plan and apply as usual from that directory. Keep the
override file out of your project's version control.

## Inspect With OpenSearch And Dashboards

The full profile also starts OpenSearch — useful as a local log or document
sink while developing:

```bash
curl -fsS "$OPENSEARCH_URL/_cluster/health" | python3 -m json.tool
curl -fsS -X POST "$OPENSEARCH_URL/app-logs/_doc" \
  -H 'Content-Type: application/json' \
  -d '{"level":"info","msg":"first event","ts":"2026-06-13T10:00:00Z"}'
curl -fsS "$OPENSEARCH_URL/app-logs/_search?q=msg:first" | python3 -m json.tool
```

Open `$DASHBOARDS_URL` in a browser, create an index pattern for `app-logs`,
and browse documents in Discover. Security plugins are disabled — it is a
local development instance, not a place for real data.

## Cleanup

Emulator state (buckets, queues, functions) lives in a local volume.
`./engine/fog down` keeps it; `./engine/fog down --volumes` removes it. See
the [FAQ](faq.md) for the full persistence picture.
