Nó là gì

Endpoint mang dáng AWS, ngay trên máy bạn

fogstack cho một dự án cá nhân một cụm Kubernetes, một image registry, một cơ sở dữ liệu Postgres, và tùy chọn thêm một API tương thích AWS cùng OpenSearch — để bạn phát triển và nối dây tích hợp mà không tốn tiền cloud hoặc chạm vào tài khoản thật.

Luật duy nhất
Một lệnh fogstack không bao giờ được đọc hoặc ghi ~/.kube hay ~/.aws của bạn, và không bao giờ được chạm tới tài khoản AWS thật. Toàn bộ thiết kế còn lại đi theo luật này.
Một công cụ phát triển cục bộ — không phải môi trường tương đương production.

Cô lập host · trái tim của thiết kế

Mọi client đều được chuyển hướng khỏi host của bạn

Trước khi bất kỳ lệnh nào chạy client kubectl, aws, helm, hoặc terraform, guard sẽ viết lại nơi từng client tìm cấu hình:

KUBERNETES

KUBECONFIG → .state/

Trỏ vào kubeconfig nằm trong repo. Host context của bạn không bao giờ bị đọc hoặc đổi — fogstack không chạy kubectl config use-context.

FILE CẤU HÌNH AWS

config & credentials → .state/

AWS_CONFIG_FILE và AWS_SHARED_CREDENTIALS_FILE trỏ vào .state/, không trỏ tới ~/.aws.

CREDENTIALS

giả, chỉ cục bộ

Export test / test, region us-east-1, cho riêng tiến trình. Endpoint instance metadata bị tắt.

BIẾN TÀI KHOẢN THẬT

bị gỡ bỏ

AWS_PROFILE, AWS_SESSION_TOKEN, AWS_ROLE_ARN bị unset. fog doctor fail nếu vẫn còn profile được set.

Cô lập host

Chắc chắn, không phải best-effort

01 · MỌI ENTRY POINT

Guard chạy ở mọi nơi

Nó không chỉ nằm trong script fog chính — các script tạo cluster và khởi động load balancer cũng source guard này. Không code path nào chạm tới client khi cấu hình host còn active.

02 · SAU KHI LOAD .ENV

Guard được khẳng định lại

Script fog load .env của bạn, rồi source guard lần nữa. Kể cả một .env cố set KUBECONFIG hoặc path AWS cũng không thể kéo client quay lại file trên host của bạn.

fog doctor in ra các check fog endpoints cho thấy file đang dùng mọi port chỉ bind 127.0.0.1

Luồng request

Một lệnh đi tới endpoint cục bộ như thế nào

1

fog up

Khởi động hoặc tái sử dụng container cho profile đã chọn và chờ healthy.

2

eval "$(fog endpoints)"

Export các biến kết nối vào shell hiện tại của bạn.

3

tool của bạn kết nối

kubectl, helm, psql, aws, và Docker dùng các biến đó để tới container cục bộ.

lưu ý — AWS SDK và CLI cần được truyền endpoint rõ ràng (--endpoint-url "$AWS_ENDPOINT_URL"); chúng không tự phát hiện endpoint. Biến là theo từng shell, nên hãy chạy lại eval trong mỗi terminal mới.

Mạng

Ba chi tiết nối dây đáng biết

IMAGE → CLUSTER

Bạn push tới localhost:5001. Registry cũng nằm trên mạng kind, và mỗi node có một containerd mirror — nên node pull cùng tag qua mạng đó.

AWS API TỪ POD

Pod gọi thẳng fogstack-emulator:4566 — không đi vòng qua host. Emulator giữ Docker socket để launch backing container của nó. OpenSearch được bridge tương tự; Postgres & Dashboards chỉ mở phía host.

LOAD BALANCER

cloud-provider-kind theo dõi Service LoadBalancer và chạy một container LB cho mỗi service, publish một port localhost. Trên Docker Desktop, published port đó là đường vào đáng tin cậy.

Port localhost được publish rất quan trọng vì ingress IP được cấp thường không route trực tiếp được từ host trên Docker Desktop.

Profile

Mặc định là minimal, dùng full khi cần AWS

MINIMAL · trường hợp phổ biến

Kubernetes, registry, database

Bộ thành phần mà hầu hết quá trình phát triển cần, và khởi động nhanh. Đây là thứ fog up launch theo mặc định.

FULL · khi có AWS trong cuộc

+ AWS API, OpenSearch, Dashboards

Thêm các dịch vụ mang dáng AWS, tốn thêm bộ nhớ và thời gian khởi động. Chuyển bằng --profile full.

vì sao tách — giữ trường hợp phổ biến thật nhanh, và không bao giờ quảng bá endpoint chết: fog endpoints cố ý bỏ AWS_ENDPOINT_URL trong chế độ minimal.
